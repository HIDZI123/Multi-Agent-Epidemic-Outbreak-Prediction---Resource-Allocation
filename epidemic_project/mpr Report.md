# MPR Report: Gaussian Mixture Models (GMM) for Hospital Operations and Patient Flows

## 1. Executive Summary
This report explains the use of Gaussian Mixture Models (GMM) in the project workflow for hospital operations under epidemic pressure. The method is applied to identify latent patterns in patient arrivals and hospital stress states, then convert those patterns into proactive resource pre-allocation and operational action plans.

In this project, GMM supports three goals:
1. Discover multiple patient arrival behaviors rather than assuming a single pattern.
2. Detect hospital stress modes as dynamic operational states.
3. Enable intelligent pre-allocation decisions before overload occurs.

## 2. What Is GMM?
A Gaussian Mixture Model is a probabilistic clustering approach that represents data as a weighted combination of multiple Gaussian distributions.

Instead of assigning each point to one rigid cluster boundary, GMM models uncertainty and overlap between clusters. This is valuable in hospital systems where operational states are not perfectly separated.

Mathematically, the density is modeled as:

$$
p(x) = \sum_{k=1}^{K} \pi_k \mathcal{N}(x \mid \mu_k, \Sigma_k)
$$

Where:
1. $K$ is the number of mixture components.
2. $\pi_k$ is the mixture weight for component $k$.
3. $\mu_k$ and $\Sigma_k$ are the mean and covariance of component $k$.

## 3. Why GMM Is a Good Fit for This Project
Epidemic hospital behavior is multi-modal:
1. Patient arrivals can have morning, daytime, and late-evening peaks.
2. Hospital operations can transition from stable to overloaded states with fuzzy boundaries.
3. Similar utilization values can represent different contexts depending on admissions and turn-away rates.

GMM is appropriate because it:
1. Captures multi-peak behavior in arrivals.
2. Handles overlapping operational regimes better than hard clustering.
3. Supports interpretable mode definitions for downstream policy logic.
4. Works well with small to medium tabular datasets generated by simulation.

## 4. Data Inputs Used in the Notebook
The notebook consumes `healthcare_epidemic_simulation.csv` and uses these key signals:
1. `day`
2. `new_infections`
3. `hospitalized`
4. `hospital_occupancy`
5. `patients_turned_away`
6. `healthcare_utilization`
7. `system_overloaded`

When event-level admission timestamps are unavailable, the workflow estimates admissions and synthesizes hourly intake patterns from daily dynamics.

## 5. Project Workflow Integration
The GMM integration in the notebook follows this sequence:

### Step A: Admission-Time Construction
1. Use real admission timestamps if present.
2. Otherwise estimate admissions from:
   - Positive change in `hospitalized`
   - `patients_turned_away`
3. Build hourly arrival time series and cyclical hour features (`hour_sin`, `hour_cos`).

### Step B: GMM for Arrival Behaviors
Features used:
1. `hour_sin`
2. `hour_cos`
3. `arrivals`
4. `arrivals_rolling_6h`
5. `healthcare_utilization`

Outcome:
1. BIC-guided selection of component count.
2. Cluster labels interpreted as flow regimes such as low, moderate, high, and surge windows.

### Step C: GMM for Hospital Stress Modes
Features used:
1. `healthcare_utilization`
2. `hospital_occupancy`
3. `hospitalized`
4. `patients_turned_away`
5. `new_infections`
6. `estimated_admissions`
7. `system_overloaded_num`

Outcome:
1. BIC-guided stress clustering.
2. Ordered stress score and mode labels such as:
   - Stable Operations
   - Rising Pressure
   - High Stress
   - Critical Overload
   - System Breakdown

### Step D: Predictive Pre-Allocation
1. Build mode transition tendencies from historical sequence.
2. Predict likely next operational mode.
3. Map expected admissions and mode severity to pre-allocation outputs:
   - beds
   - nurses
   - ICU slots
   - PPE kits

### Step E: Step 7 Operational Decision Engine
1. Translate prediction outputs into alert levels:
   - Normal
   - Watch
   - Warning
   - Critical
2. Generate actionable plan bundles for staffing, triage, communication, and surge response.
3. Export operational action plan as a deployable artifact.

## 6. Significance to This Project
This GMM layer is significant because it shifts the project from descriptive epidemic simulation to proactive decision intelligence.

Key impact areas:
1. Earlier risk detection.
   - Reveals latent stress states before binary overload indicators trigger.
2. Better resource efficiency.
   - Enables right-sizing resources by expected behavior mode.
3. Stronger operational readiness.
   - Converts technical predictions into practical hospital actions.
4. Multi-agent compatibility.
   - Produces structured outputs that can be consumed by hospital, triage, logistics, and policy agents.
5. Improved resilience under volatility.
   - Handles uncertainty and overlap inherent in outbreak dynamics.

## 8. Assumptions and Limitations
1. Synthetic admission-time generation is an approximation when real timestamps are unavailable.
2. Transition logic is first-order and may not capture long-memory effects.
3. Mode labels are interpretable summaries, not medical diagnoses.
4. Cluster quality depends on data representativeness and feature engineering.

## 9. Recommended Enhancements
1. Replace synthetic admission timing with real event-level intake data.
2. Calibrate alert thresholds using domain expert feedback and retrospective events.
3. Add temporal models (HMM or sequence models) for richer mode transition prediction.
4. Introduce uncertainty-aware allocation ranges rather than single-point recommendations.
5. Validate outcomes against historical surge periods and operational KPIs.

## 10. Conclusion
GMM provides a strong probabilistic foundation for identifying patient-flow behaviors and hospital stress modes in this epidemic resource-allocation project. Its integration enables actionable, mode-aware planning and supports the evolution from simulation to intelligent, anticipatory hospital operations.
